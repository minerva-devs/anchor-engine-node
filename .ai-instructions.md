## 🚨 CRITICAL RULES FOR AI ASSISTANTS

### Rule #1: Documentation Location (MANDATORY)

**NEVER create documentation outside these allowed directories:**

```
✅ ALLOWED:
- docs/           - All user-facing documentation
- specs/          - Technical specifications and standards
- [PACKAGE]/README.md - Package-specific READMEs only (e.g., mcp-server/README.md)

❌ FORBIDDEN:
- Root directory (*.md files at project root)
- Random subdirectories (scripts/, tests/, benchmarks/, etc.)
- Hidden directories (.ai/, .cursor/, etc.)
```

**Before creating ANY .md file:**
1. Ask: "Does this need to be documentation?"
2. If yes → Create in `docs/` or `specs/` ONLY
3. If it's a package README → Keep in package root only

---

### Rule #2: No Duplicate Documentation

**NEVER duplicate content that already exists:**

- Check `docs/` before writing
- Check `specs/` before writing
- Link to existing docs instead of rewriting
- Update existing docs instead of creating new ones

**Before writing:**
```bash
# Search for existing content
grep -r "your topic" docs/ specs/
```

---

### Rule #3: Documentation Must Be Instructional

**ONLY create documentation that:**
- ✅ Teaches how to do something
- ✅ Explains a system/architecture
- ✅ Documents an API or interface
- ✅ Provides troubleshooting guidance

**NEVER create:**
- ❌ "About" pages with no actionable content
- ❌ Duplicate setup guides
- ❌ Philosophical discussions (use specs/ for decisions)
- ❌ Meeting notes or temporary scratchpads

## 🔍 Before You Create Documentation

**Checklist:**
1. [ ] Does this content already exist in `docs/` or `specs/`?
2. [ ] Is this the correct location (`docs/` vs `specs/`)?
3. [ ] Is this instructional (not philosophical)?
4. [ ] Is this concise (not verbose)?
5. [ ] Am I creating this in an allowed directory?

**If ANY answer is NO → Stop and ask the user.**
